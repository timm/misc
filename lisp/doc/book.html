Small AI, by Design

Explainable multi-objective optimization, in four hundred lines of code.

When big AI has a verification problem, use small AI. Here, we show a few hundred lines of code that do the AI that actually matters (for decision making; for multi-objective optimization, classification, regression) all from one active-learning loop that needs just a few dozen labels. Explainable decision trees you can print on a page. Honest train/test evaluation with a pay-to-play holdout budget. One file, no dependencies, milliseconds to load and execute: every line fits on one screen.

Prologue — A history of less

Email is a wonderful thing for people whose role in life is to be on top of things. But not for me; my role is to be on the bottom of things.
— Donald Knuth1

Every era of AI added machinery to compensate for something scarce. We now have enough of everything that most of the machinery can come back off. This is a book about taking it off.

The eleven-step arc, each step adding a layer, each layer paying for what was missing:

  1. Greeks. Rationality as certainty. Nothing scarce except empirical grounding; deduction alone cannot handle the world.
  2. Gödel (1931). Rationality becomes an empirical study. The dream of completeness ends; reasoning joins the other sciences.
  3. Simulated annealing (Kirkpatrick et al., 1983). Sampling from one distribution converges on a canonical other. Memory is scarce: one walker, a temperature.
  4. Genetic algorithms (Holland, 1975; 80s). Populations instead of one walker. Memory is cheaper now.
  5. Knowledge era. We used knowledge to constrain and speed inference. Expert systems, priors, pruning. Brittle, but focused.
  6. Local search (Selman et al., AAAI-92; Luby et al., 1993). CPU is cheap; restarts dominate. Finding in our own case studies: restarts matter more than the mutation policy. Simulated annealing and local search differ by three lines.
  7. Big-data era. Data got cheaper than compute. Problems grew larger to match.
  8. Surrogate era. Approximate the oracle with a fast model, reason against the surrogate instead.
  9. Domingos (1997) & arXiv:1702.05701 (2017). A bad predictor can still be a good optimizer. The accuracy cult was a detour.
  10. Treatment learning. Drop the optimizer. Play snakes-and-ladders with the surrogate: contrast sets between high- and low-value outputs.
  11. ezr. All we need is to walk the landscape of the data used to build the surrogate.

Arc 1 is the long road. Arc 2 — Pearl's ladder of causation — is what's left when you don't need arc 1 anymore. The rest of this book is arc 2, with arc 1 acknowledged in the margins.

Sutton was half right. Yes — discard your a priori priors. No — you don't need to pay compute scale for that. A handful of ranges learned from the data is a valid learned prior that obeys Sutton's rule one without endorsing rule two. The bitter lesson was about whose knowledge, not about how much compute. Compute scale got smuggled in.

Seven themes follow. Each has a demo you can run in under a second:

  1. The landscape nobody looks at.
  2. Under-the-hood sameness: k-means, SA, Bayes, trees are the same substrate.
  3. Causation is easy: the three-rung ladder as three tree walks.
  4. You don't need much data: fifty labels beat the big-data baseline.
  5. Your leaderboard is mostly noise.
  6. Deltas are simpler than the things.
  7. Agents yes; LLMs all the way down, no.

1. The landscape nobody looks at

The field is called data mining. Most textbooks ignore the data.

They race to algorithms. Decision trees, gradient descent, backprop, transformers. Each one works only because the input has regularities: correlations, clusters, separable classes, distributions that repeat themselves. If rows were random noise, no algorithm would help. So what shape is the data?

Two regularities matter. First, values have types. Some are numbers; some are words. You cannot add "five" and "three" but you can add 5.0 and 3.0. You cannot ask for the mean of ["red", "green", "blue"], but the mode is fine. Two struct-like constructors cover it:

function Sym(s,n)
  return l.new(SYM,{txt=s or "",at=n or 0,has={},n=0}) end

function Num(s,n)
  return l.new(NUM,{txt=s or "",at=n or 0,n=0,mu=0,m2=0,
                    heaven=s and s:match"-$" and 0 or 1}) end

Both carry a position (at), a name (txt), a count (n). A Sym keeps a frequency dictionary. A Num keeps Welford's running mean and sum of squared deviations. heaven is the direction of good: 1 for maximize, 0 for minimize.

Second regularity: names carry roles. Capitalize numerics. Suffix goals with + or -. Mark a classifier target with !. Mark skipped columns with X. The classifier is eight lines:

function Cols(names,    xs,ys,all,col,klass)
  xs,ys,all,klass = {},{},{},nil
  for at,s in ipairs(names) do
    col = l.push(all,(s:match"^[A-Z]" and Num or Sym)(s,at))
    if s:match"!$" then klass = col end
    if not s:match"X$" then
      l.push(s:match"[%+%-!]$" and ys or xs, col) end end
  return l.new(COLS,{x=xs,y=ys,all=all,klass=klass,names=names}) end

Convention over configuration: the data tells us what the data is. No registry, no manifest, no plugin file.

Stats live in the column, not a separate DataFrame

One idea drives the design: the stats are the data. There is no DataFrame of statistics alongside a list of rows. Every column object carries its own running summary. Ask for the mean; read mu. No separate fit step.

For a Num, Welford's online update keeps mu and the helper m2 in O(1) per sample:

function NUM._add(i,v,w,    d)
  i.n = i.n + w
  if w<0 and i.n<=1 then i.n,i.mu,i.m2 = 0,0,0
  elseif i.n > 0 then
    d = v-i.mu
    i.mu = i.mu + w*d/i.n
    i.m2 = i.m2 + w*d*(v-i.mu) end end

Pass w=-1 and the same formulas unwind a previous add. Rows can be discarded after they pass through; the Num remembers what matters. This is single-source-of-truth for the column. The summary is the stored form.

Normalize via sigmoid-of-z

Two columns often have different scales. Normalize each to [0, 1]:

function NUM.norm(i,v,    z)
  if v=="?" then return v end
  z = (v - i.mu)/(i:spread() + 1e-32)
  return 1/(1 + exp(-1.7*l.crop(z,-3,3))) end

A z-score, clamped to [-3, 3], through a logistic squash — a standard approximation to the Gaussian CDF. Range-free and outlier-tolerant. For a Sym, norm is the identity; equality is the only comparison that matters.

Small prior art on not-looking-at-data

Kohavi & John (1997) already showed that a 100-feature dataset often reduces to 5–10 features without accuracy loss.6 Williams, Gomes & Selman (2003) showed that hard SAT instances usually have small backdoor variable sets — a handful of controllers for a huge system.7 The Johnson–Lindenstrauss lemma (1984) says high-dimensional points can be projected to much lower dimensions with bounded distortion.8 The manifold/semi-supervised wave (Isomap, LLE, Laplacian eigenmaps, ~2000–2005) generalized this to non-linear structure.9

Four decades of results saying the same thing. Most of your data is decoration. The signal is in a handful of columns and a handful of ranges. You don't need a deep net to find them. You need to look.

Why Lua?

Three reasons, stated loudly:

1. Succinct and Python-like. Whitespace-insensitive but feels like Python. One data structure: the table. First-class functions. String pattern matching as a built-in. That is almost the whole language.

2. Fast where it matters. Measured on auto93.csv (--active, one CSV load, 20 internal train/test splits): stock Lua 5.4 does the algorithm work in 54 ms, starts in 2 ms. Python on the same workload does the work slightly faster (44 ms) but starts 7× slower (15 ms), so Lua wins wall-clock. Julia loses badly (4.8 seconds of JIT recompile, even with --compile=min -O0). A Crystal port (ahead-of-time compiled) does the work in 2.8 ms — 19× faster than Lua, 16× faster than Python — but costs you static types. Lua's pitch here is not "fastest"; it is "fast enough, and readable." See the Epilogue for the full table.

3. Pedagogy in the age of LLMs. The exercises in this book are translations: "here's the Lua, write the Python." LLMs will help you. Good — that's reading comprehension via production. But exams test code details, not code generation. You will be asked what a seven-line function does, line by line. So use an LLM to explore and to explain. Do not use one to avoid reading. Lua is unfamiliar enough that you cannot skim LLM output and nod along; you must verify. That friction is the point.

2. Under-the-hood sameness

You already know some of this. K-means is cluster-by-centroid. Simulated annealing is search-by-accept-function. Naive Bayes is classify-by-column-likelihood. Decision trees partition-by-split-score.

Four algorithms, four textbooks. One substrate.

Every one of them reduces to: keep a running summary of columns (Num for numbers, Sym for symbols), ask for a center or a spread, compute a distance or a likelihood, iterate. That is it. The substrate was built in Chapter 1.

Same polymorphism, no OO ceremony

Ask a column for its mid and spread. A Num answers with Welford's mean and stddev. A Sym answers with the mode and Shannon entropy. Same call, type-specific answer. The caller never writes if getmetatable(col) == NUM:

function NUM.mid(i) return i.mu end
function SYM.mid(i,    most,mode)
  most = -1
  for v,n in pairs(i.has) do if n>most then most,mode = n,v end end
  return mode end

function NUM.spread(i)
  return i.n>1 and (max(0,i.m2)/(i.n-1))^0.5 or 0 end
function SYM.spread(i)
  return -l.sum(i.has,function(v) return v/i.n*log(v/i.n,2) end) end

Distance is Minkowski with one knob

Manhattan, Euclidean, Chebyshev: the same algorithm at p = 1, 2, ∞. One function covers all three:

function l.minkowski(t,fn,    pw)
  pw = function(c) return fn(c)^the.p end
  return (l.sum(t,pw)/#t)^(1/the.p) end

function DATA.disty(i,row)
  return l.minkowski(i.cols.y, function(c)
    return abs(c:norm(row[c.at]) - c.heaven) end) end

function DATA.distx(i,r1,r2)
  return l.minkowski(i.cols.x, function(c)
    return c:aha(r1[c.at],r2[c.at]) end) end

Two lines of logic. Swap -p 1 for -p 2; neighbors shuffle; nothing else changes.

Bayes is a contrast

Classify a row by which class it is likelier under. Same Num/Sym substrate:

function l.like(c,v,prior,    sd)
  if v=="?" then return 1 end
  if c.has then
    return ((c.has[v] or 0) + the.k*prior)/(c.n + the.k) end
  sd = c:spread() + 1e-32
  return (1/(2*pi*sd*sd)^0.5) * exp(-(v-c.mu)^2/(2*sd*sd)) end

Bayes reappears in Chapter 6 as a contrast tool: "row likelier under best-Data or under rest-Data?" Same code, different framing.

Search is the accept function

Simulated annealing and local search share the same (1+1) scaffold. The two differences are the accept rule and the neighborhood:

-- SA: stochastic accept (Boltzmann), mutate half the x-columns at once.
if en < e or rand() < exp((e-en)/(1-h/budget+1e-32)) then s, e = sn, en end

-- LS: greedy accept; pick ONE column and try many values of it before moving on.
if en < e then s, e = sn, en end

SA explores breadth-first across dimensions; LS exploits one dimension at a time. Restart policy is optional for both — stagnate for N steps without improvement, start over from a new random row.

Empirical finding: restarts matter for LS, not SA

Running all four combinations on auto93 (lua ezr.lua --compare, 20 repeats, 500 oracle calls per run, nearest-neighbor surrogate trained on 50 known rows):

# best   mid  worst    sd  runs  tied  name
# ----  ----  -----  ----  ----  ----  ----
  0.07  0.12   0.19  0.03    20     *  sa-r
  0.07  0.12   0.19  0.04    20     *  sa+r
  0.07  0.13   0.22  0.04    20     *  ls+r
  0.07  0.19   0.33  0.08    20        ls-r    ← NOT tied
  0.12  0.28   0.51  0.11    20        rand

Three of the four tie statistically. The loser is LS without restart: median 0.19, worst-case 0.33. The other three sit near 0.12 median. SA with or without restart ties, because SA's stochastic accept is already its own escape mechanism. LS is greedy; if it gets stuck, only a restart rescues it.

This is a sharper claim than "restarts dominate mutation": restarts only matter when the search has no other escape. Selman (AAAI-92) and Luby et al. (1993) proved optimal restart schedules for SAT thirty years ago1011; the same principle applies here. Add one of {stochastic accept, multi-column mutation, restarts}; you only need one.

Beware the median-only leaderboard. Chen et al. (2022) argue that reporting only medians hides the fragility of stuck runs — a point visible above in the worst column: sa-r's worst (0.19) is almost as good as sa+r's (0.19), but ls-r's worst (0.33) blows up.17 Report the best, the median, and the worst. Let the reader see the spread.

GA is a sidebar

Genetic algorithms are SA with a population. Same accept function, more memory. We did not implement GA because nothing in this book is memory-bound. If you have tens of thousands of cores and no memory pressure, GA wins. You probably don't.

Clustering is centroid-over-Num

K-means assigns each row to the nearest centroid, recomputes each centroid as the mean of its assigned rows, iterates. The centroid is a vector of mid values over the Data's columns — which is already defined:

function DATA.mid(i)
  i._mid = i._mid or l.map(i.cols.all,function(c) return c:mid() end)
  return i._mid end

Clustering is thus composition: Data + distx + mid + reassign. Nothing new. Recursive halving (rhalf), random-projection clustering, kpp seeding — all variations on who gets called in what order. Not new algorithms; new orderings of the same substrate.

3. Causation is easy — the three-rung ladder

Pearl distinguishes three kinds of question:

  • Rung 1 — Association. What is happening?
  • Rung 2 — Intervention. What happens if I change something?
  • Rung 3 — Counterfactual. What should I have done differently?

Most AI work stays on Rung 1. Pearl argues Rung 3 is where decisions live. He also argues Rung 3 is hard. For opaque models it is. For a tree, Rung 3 is a loop.

One tree, three walks

Rung 1 — print the tree (--see). The explanation is the model. No post-hoc SHAP, no LIME:

Rung 2 — run rows down the tree; flag leaves that disagree with reality (--act). A leaf predicts a y-value; a row's real y can be compared; when they diverge by more than the leaf's spread, that row is a surprise. Surprises are the signal: the next rows to ask the oracle about.

function TREE.leaf(i,row,    v,ok)
  if not i.col then return i end
  v = row[i.at]
  if v=="?" then return i.left:leaf(row) end
  ok = i.col.mu and (v<=i.cut) or (v==i.cut)
  return (ok and i.left or i.right):leaf(row) end

Rung 3 — diff the leaves (--imagine). For any current leaf, find other leaves with lower mid-y and report the x-column values that differ. That is a counterfactual plan:

function TREE.plan(i,here,    out,eps,dy,diff)
  out, eps = {}, the.gap * here.y:spread()
  i:nodes(function(n)
    if not n.col and n ~= here then
      dy = here.y:mid() - n.y:mid()
      if dy > eps then
        diff = {}
        for k,v in pairs(n.mids) do
          if here.mids[k] ~= v then
            l.push(diff, k.."="..l.o(v)) end end
        if #diff > 0 then
          l.push(out,{dy=dy, score=n.y:mid(), diff=diff}) end end end end)
  return l.sort(out, function(a,b) return a.dy > b.dy end) end

Fifteen lines. Pearl's Rung 3 in fifteen lines. Not a transformer. A tree traversal.

Why this is cheap

Naive Rung 3 would search input space — combinatorial. The tree already discovered a finite set of plausible regions (the leaves). Rung 3 walks among those, not among raw values. That is step five of the prologue arc (knowledge constrains inference) paying the rent for step eleven (walk the landscape). The tree is the knowledge. The leaves are the hypothesis space. The jump is the counterfactual.

A case study: BI collapses to the ladder

Buse & Zimmermann (ICSE 2012, fig. 6) tabulate the information needs of software analytics: twenty tools across three task categories — monitoring, diagnosis, prescription.12 Both axes collapse to Pearl's three rungs:

  • Monitor ≡ Rung 1 ≡ treeShow.
  • Diagnose / alert / what-if ≡ Rung 2 ≡ treeLeaf vs. reality.
  • Recommend / plan / prescribe ≡ Rung 3 ≡ treePlan.

Twenty tools become three rungs. Three rungs become three walks of one tree. An industry's product catalog, compressed to a schema.

4. You don't need much data

Labels cost money. An expert takes minutes to hours per annotation. A fuzzer run takes compute. A log review takes attention. Four hundred labels is a two-day task.

So pick carefully. Do not label everything. Do not label randomly. Label the rows where labeling will teach you something.

The filter

Keep two labeled subsets: best and rest. For each unlabeled candidate, ask: is it closer to best's centroid or to rest's?

function closer(lab,best,rest,r,t)
  return lab:distx(r,best:mid()) < lab:distx(r,rest:mid()) end

Two distx calls and a less-than. That is the acquisition heuristic. Alternatives exist (Bayesian information gain, expected model change); this is the simplest that works.

Schedule: explore then exploit

The active loop passes a progress dial t ∈ [0, 1] to the scorer on each iteration. Early (t low), a scorer can favor exploration; late (t high), exploitation. One number, no new algorithm:

for k = 5, min(the.few, #rs) do
  if ys.n >= the.Budget then break end
  t = ys.n / the.Budget
  if score(lab,best,rest,rs[k],t) then
    add(ys, lab:disty(rs[k]))
    add(lab, rs[k]); add(best, rs[k])
    rebalance(best,rest,lab) end end

Want Bayes-based acquisition instead of centroid-based? Pass acquireBayes as the score argument. Same loop, different scorer. That is the "under-the-hood sameness" of Chapter 2 doing its job.

Self-correction: rebalance

Newly-added best rows sometimes turn out mediocre. Cap |best| at √|lab| and evict its worst on overflow:

function rebalance(best,rest,lab,    bad,badD,d)
  if #best.rows > (#lab.rows)^0.5 then
    badD = -1E32
    for _,r in ipairs(best.rows) do
      d = lab:disty(r)
      if d>badD then bad, badD = r, d end end
    sub(best,bad); add(rest,bad) end end

Reversible add/sub makes this cheap: no rebuild, just recompute the delta.

Does a bad tree matter?

The tree we build on the labels need not be accurate in an absolute sense. It only needs to rank leaves correctly. Domingos (1997) and Nair et al. (2017) showed this formally for bagging and for configuration optimization respectively.13 14 A bad predictor is often a fine optimizer because optimization cares about order, not calibration.

This is why fifty labels suffice. The tree is noisy at small N, but its ranking of holdout rows is usable even when its predictions are not.

5. Your leaderboard is mostly noise

Twenty algorithms. One winner. Announce.

Only: the runs vary. Seeds matter. A two-point lead across ten trials might be real, might be luck. The median-of-medians leaderboard is a tradition in ML papers; it is also, often, a lie.

Group by tie

Cliff's Delta measures ordinal dominance. The two-sample Kolmogorov–Smirnov test measures distribution separation. Both under thresholds → statistically indistinguishable. Both over → different. ezr composes the two into same:

function l.same(xs,ys,eps,    n,m,gt,lt,d,ks,fa,fb)
  xs, ys = l.sort(l.slice(xs)), l.sort(l.slice(ys))
  n, m = #xs, #ys
  if abs(xs[n//2+1] - ys[m//2+1]) <= eps then return true end
  ...
  return ks <= 1.36 * sqrt((n+m)/(n*m)) end

bestRanks then groups treatments that tie with the best. The output is a set of winners, not a ranked list:

sa     (tied-best)  mid=0.23  sd=0.04  n=20
ls     (tied-best)  mid=0.24  sd=0.05  n=20

Random is not in the tied-best set here. Two other algorithms are. Announcing "SA won" would have been wrong; announcing "SA and LS tied" is honest.

Dumb baselines win more than you think

Include them. A random-sample cluster. A median-split tree. "Pick the best of five random rows." These often land in the tied-best set. If you do not include them, you will keep claiming wins that are not wins.

Run lua ezr.lua --compare auto93.csv. Read the tied-best output. How many of your recent published ML result tables would survive this test?

6. Deltas are simpler than the things

You have been reasoning about objects. You should be reasoning about differences.

Contrast set learning is this claim made concrete. A handful of attribute ranges is enough to separate best from rest. You do not need to predict y-values; you need to describe what separates high-y rows from low-y rows. The explanation is the diff.

Three-stage pipeline

Stage 1 — unsuper. Sort (x,y) pairs by x; walk the sorted list; open a new bin when the current bin is large enough and the current span is wide enough:

function l.unsuper(xys,big,iota,    now,all,x,y)
  l.sort(xys, function(a,b) return a[1]<b[1] end)
  now = {lo=xys[1][1],hi=xys[1][1],n=0,y=Sym()}
  all = {now}
  for i,xy in ipairs(xys) do
    x, y = xy[1], xy[2]
    if i < #xys - big and now.n >= big
       and (now.hi - now.lo) > iota and x ~= xys[i+1][1] then
      now = {lo=x,hi=x,n=0,y=Sym()}
      l.push(all, now) end
    now.n = now.n + 1
    now.hi = x
    add(now.y, y) end
  return all end

Stage 2 — merge. Walk adjacent bins; combine if the merged y-spread does not exceed the weighted spread of the two halves. Repeat until fixed point:

function l.merge(b4,    tmp,j,a,b,cy)
  ...
  cy = l.mergeSym(a.y, b.y)
  if cy:spread()*0.95 <=
     (a.y:spread()*a.n + b.y:spread()*b.n)/(a.n + b.n) then
    a = {lo=a.lo, hi=b.hi, n=a.n+b.n, y=cy}
    ... end
  return #tmp < #b4 and l.merge(tmp) or b4 end

Stage 3 — WHICH. Conjoin ranges into rules. Score rules by b²/(b+r): support times probability-of-best. Grow a stack by rank-weighted mating:

function l.which(best,rest,cols,    stack,a,b,kid,s,rng)
  stack = {}
  for _,rng in ipairs(l.seedRanges(best,rest,cols)) do
    l.push(stack,{combo={rng},
                  s=l.whichScore({rng},best.rows,rest.rows)}) end
  l.sort(stack, function(a,b) return a.s > b.s end)
  for _ = 1, the.gens do
    a, b = l.pick(stack), l.pick(stack)
    kid = {}
    for _,rng in ipairs(a.combo) do l.push(kid,rng) end
    for _,rng in ipairs(b.combo) do l.push(kid,rng) end
    s = l.whichScore(kid,best.rows,rest.rows)
    l.push(stack,{combo=kid,s=s})
    l.sort(stack, function(a,b) return a.s > b.s end)
    while #stack > the.cap do stack[#stack] = nil end end
  return stack[1] end

About sixty lines for the whole pipeline. The output is a human-readable rule like Mpg in [22, 32] AND Cyl == 4, scored by how well it separates best from rest.

Connection to everything else

  • Naive Bayes as a contrast tool: likes(best, r) > likes(rest, r). Same idea, different representation.
  • Counterfactual planning (Chapter 3, Rung 3): the "diff" is a contrast set between two leaves.
  • Treatment learning (TAR2, Menzies & Hu 2003): this book's direct ancestor.15
  • STUCCO (Bay & Pazzani 2001): the canonical contrast-set mining algorithm.16

Try it yourself

Run lua ezr.lua --which auto93.csv. Count the ranges in the top rule. Usually two or three. Compare with the number of parameters in any ML model you respect. Ask whether the extra parameters earn their place.

7. Agents yes; LLMs all the way down, no

The future is not one giant model orchestrating itself. It is a small model orchestrating a toolbox of cheap, auditable workers.

An LLM cannot run Welford's update. It cannot retract a row. It hallucinates the median. But: it is excellent at task decomposition, at reading a tree and summarizing it in prose, at asking a clarifying question, at dispatching to the right tool.

The pattern is: LLM as router, ezr as engine.

  • User: "what if we change the engine size?"
  • LLM: dispatch to TREE.plan with here as current leaf; summarize the output.
  • ezr: returns three counterfactual rules, scored.
  • LLM: "changing Mpg to [22, 32] and Cyl to 4 would move you from 0.41 to 0.28."

The LLM is ~30 lines of glue. The engine is ezr. The whole thing runs on a laptop. It is reproducible under a seed. It is auditable (the rules are printable).

The alternative — all-LLM agent stacks calling LLMs to reason about LLM-predicted data — is expensive, opaque, and non-reproducible. We do not recommend it.

Why now

The same LLMs that made every task look like an LLM task also made tool use reliable enough that non-LLM tools are trustworthy subagents. You can delegate statistics to ezr and language to the LLM. The LLM becomes the user interface; ezr becomes the ground truth.

This is step eleven of the prologue arc (walk the landscape) done via step five (knowledge constrains inference) with the LLM playing orchestrator (step six, restarts — the LLM can always try again). Each piece earns its place.

Try it yourself

Wire your favorite LLM to ezr's library. Expose classify, active, which, TREE.plan as tools. Ask it to explain a decision from auto93.csv. Compare the cost and reproducibility to doing the whole thing in the LLM.

Epilogue — We told you so

Knuth chose to be on the bottom of things. Fifteen hundred years before him, the Greeks tried to reason from certainty. Gödel told us certainty was not on offer. We compensated with memory, then populations, then knowledge, then restarts, then data, then surrogates, then accepted that surrogates don't need to be accurate, then realized we could drop the optimizer altogether, then realized all we needed was to walk the landscape of the data the surrogate was going to be built on.

Arc 1 is how we got here. Arc 2 is what's left after you don't need arc 1 anymore.

If you cannot see the whole algorithm on one screen, you do not understand it. You trust it. Trust is a bug.

Appendix A — Lua for the curious

Tables, and nothing else

There is one data structure: the table. It is a dict and a list simultaneously. The "array part" is integer-keyed, zero overhead; the "hash part" is everything else. #t is the length of the array part; pairs walks both parts, ipairs walks only the array part up to the first nil.

Metatables (and why your discomfort is misplaced)

A metatable tells another table how to behave. The most useful entry is __index: "on missing field, look here instead." Set NUM.__index = NUM and every Num instance falls back to NUM when a method is not on the instance. That is OO dispatch in one line.

function l.new(kl,obj) kl.__index = kl; return setmetatable(obj,kl) end

Two oddities, both harmless:

Self-reference. kl.__index = kl. Reads weird, does exactly the right thing: "if a method is not on the instance, look on the class." The self-reference is not a bug; it is the dispatch rule stated as a field.

Repeated assignment. l.new sets kl.__index on every call. Redundant but idempotent. You could hoist it:

local NUM = {}; NUM.__index = NUM

and then new would only call setmetatable. Same behavior, one less assignment per constructor. We do not bother because the cost is tiny and the factoring is uniform across all five classes.

Mental model: a metatable is Lua's __getattr__; __index = CLASS makes missing-field lookup fall back to CLASS; that is method dispatch.

Forward-declared locals

Lua resolves function names at call time, but a local declaration must exist before any reference. Mutually recursive functions need their names declared first:

local Tree, Sym, Num, Cols, Data
local add, sub, adds, wins, split
local closer, rebalance, active, validate, acquireBayes

Then the function definitions reassign those locals. Without the forward declarations, one function cannot see another defined later in the file.

The four-space gap in parameter lists

A convention, not a language feature:

function Cols(names,    xs,ys,all,col,klass)

Four spaces separate actual parameters (before) from scratch locals (after). The scratch vars are initialized in the body; they are parameters only so that they are local, not global. Lua has no let form; this is the idiom.

String patterns, not regex

s:match"^[A-Z]" checks whether s starts with uppercase. The syntax is Lua's pattern language, which is not full regex — no alternation, limited character classes — but covers ninety percent of parse-tasks in thirty percent of the characters. The lack of parens on a single string-literal argument is legal Lua.

Iterators via closures

Our CSV reader yields one typed row at a time:

function l.csv(src,    f,id)
  f = assert(io.open(src), "cannot open "..tostring(src))
  id = 0
  return function(    s,t)
    s = f:read()
    if s then
      t = {}; id = id + 1; t.id = id
      for x in s:gmatch"[^,]+" do
        l.push(t, l.thing(x:match"^%s*(.-)%s*$")) end
      return t
    else f:close() end end end

The outer function opens the file and returns a closure. Each call to the closure reads one line or closes the file and returns nil. Lua's for r in l.csv(path) do ... end loop stops when the closure returns nil.

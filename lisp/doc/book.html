Small AI, by Design

Explainable multi-objective optimization, in four hundred lines of code.

When big AI has a verification problem, use small AI. Here, we show a few hundred lines of code that do the AI that actually matters (for decision making; for multi-objective optimization, classification, regression) all from one active-learning loop that needs just a few dozen labels. Explainable decision trees you can print on a page. Honest train/test evaluation with a pay-to-play holdout budget. One file, no dependencies, milliseconds to load and execute: every line fits on one screen.

1. Motivation

Email is a wonderful thing for people whose role in life is to be on top of things. But not for me; my role is to be on the bottom of things.
— Donald Knuth1

In an age where everyone hands details to an LLM, what are we missing?

We refactored an AI research codebase to the bone. Four hundred lines of Lua, one file, no dependencies beyond the Lua standard library, runs in 60 milliseconds. Five things stopped being true.

One. Regression, classification, and optimization are the same task. Our active-learning loop does all three. The only difference is a suffix in the CSV header. A column named Mpg- is a minimization target. Mpg+ is maximization. class! is a classifier label. Everything else is a feature. Three textbook chapters, one algorithm.

Two. Five abstractions, everywhere. Num, Sym, Cols, Data, and a distance function. Every module composes from these. The Num that tracks mean and standard deviation is one constructor of five fields. That same Num is the beating heart of the active learner, the splitting criterion for decision trees, and the basis of every "is X better than Y" judgement the code makes. Build once, reuse always.

Three. Interpretability is cheap. Rudin argued that high-stakes decisions should use interpretable models, not explained black boxes2. We agree. The cost is small. Our decision tree learner is eighty lines of Lua. Trees print on a page. We measured the tax: at thirty labels, a tree-ranked holdout loses six points of win to a centroid baseline. At fifty labels the tax is zero. Interpretability has a minimum sample size. Above it, it costs nothing.

Four. Fewer than fifty labels beat big data. Labels in software engineering are expensive. Expert annotation fatigues quickly. Historical logs are unreliable; up to ninety percent of technical-debt entries are false positives3. Automated oracles use approximate heuristics that miss the interesting cases. Active learning picks informative rows and stops. Fifty labels outperform baselines trained on thousands.

Five. Dependencies are a supply-chain bet. Modern software is eighty-five to ninety-seven percent external code4. Each dependency widens the attack surface. A single poisoned npm or PyPI update compromises thousands of downstream packages silently. Richard Hipp, author of SQLite, practices backpacking: cutting code to what is essential. He built his own editor, servers, storage engine. He shunned Berkeley DB. Oracle later acquired it and demanded license fees. SQLite ships today on every Android phone, every iPhone, Chrome, Firefox, Safari, most Linux distributions, and Mars rovers. Hipp's framing: if you want to be free, that means doing things yourself5. Our CSV reader is eight lines. Our CLI parser is fourteen. The whole file is four hundred.

Why Lua?

Lua is the "just enough" language. One data structure: the table, which is both dictionary and array. First-class functions. Metatables for polymorphism. String pattern matching as a built-in. That is almost the whole language. No class declarations, no type annotations, no JIT tax, no compile pause. A Lua script starts in 10 milliseconds.

You may not know Lua. It is the language embedded in Redis, nginx, World of Warcraft, Adobe Lightroom, Wireshark, countless games and routers and databases. Tens of millions of devices run it; most developers have never used it directly. It is ordinary once you see it: function foo(x) ... end, tables with {k=v, ...}, method syntax obj:method(arg) that desugars to obj.method(obj, arg). Learn the rest as you go. Every chapter ends with a Try it yourself list of the exercises a port would need in your language of choice.

The claim

If you cannot see the whole algorithm on one screen, you do not understand it. You trust it. Trust is a bug.

2. The landscape

The field is called data mining. Most textbooks ignore the data.

They race to algorithms. Decision trees, gradient descent, backprop, transformers. Each one works only because the input has regularities: correlations, clusters, separable classes, distributions that repeat themselves. If rows were random noise, no algorithm would help. So what shape is the data?

Two regularities matter. First, values have types. Some are numbers; some are words. You cannot add "five" and "three" but you can add 5.0 and 3.0. You cannot ask for the mean of ["red", "green", "blue"], but the mode is fine. Two struct-like constructors cover it, each wrapping a Lua table in a metatable:

function Sym(s,n)
  return l.new(SYM,{txt=s or "",at=n or 0,has={},n=0}) end

function Num(s,n)
  return l.new(NUM,{txt=s or "",at=n or 0,n=0,mu=0,m2=0,
                    heaven=s and s:match"-$" and 0 or 1}) end

Both carry a position (at), a name (txt), a count (n). A Sym keeps a frequency dictionary. A Num keeps Welford's running mean (mu) and sum of squared deviations (m2). heaven is the direction of good: 1 for maximize, 0 for minimize.

Lua has no classes and no type declarations. What l.new does is set a metatable on the table, so that method lookups (obj:method(arg)) find functions like NUM._add or SYM._add. That is our polymorphism. Single-dispatch on the receiver, done with a ten-line helper:

function l.new(kl,obj) kl.__index = kl; return setmetatable(obj,kl) end

Second regularity: names carry roles. Capitalize numerics. Suffix goals with + or -. Mark a classifier target with !. Mark skipped columns with X. The classifier is eight lines:

function Cols(names,    xs,ys,all,col)
  xs,ys,all = {},{},{}
  for at,s in ipairs(names) do
    col = l.push(all,(s:match"^[A-Z]" and Num or Sym)(s,at))
    if not s:match"X$" then
      l.push(s:match"[%+%-!]$" and ys or xs, col) end end
  return l.new(COLS,{x=xs,y=ys,all=all,names=names}) end

Capital first letter gives a Num. Lowercase gives a Sym. The last character sets the role. Convention over configuration: the data tells us what the data is. No registry, no manifest, no plugin file.

Data is rows plus their summary:

function Data(src,    d)
  d = l.new(DATA,{rows={},cols=nil,_mid=nil})
  if type(src)=="string"
  then for    r in l.csv(src)       do add(d,r) end
  else for _,r in ipairs(src or {}) do add(d,r) end end
  return d end

One idea drives the design: the stats are the data. There is no DataFrame of statistics alongside a list of rows. Every column object carries its own running summary. Ask for the mean; read mu. No separate fit step. Pop a row with sub and the summaries correct themselves.

The mutation vocabulary has four verbs. add creates, sub deletes, field-access reads (n.mu), assignment updates (n.mu = x). CRUD, the same four every REST API pretends to have. add pushes a value into a column's summary and returns the value; sub is add with weight -1:

function add(i,v,w) if v~="?" then i:_add(v,w or 1) end; return v end
function sub(i,v)   return i:_add(v,-1) end

function NUM._add(i,v,w,    d)
  i.n = i.n + w
  if w<0 and i.n<=1 then i.n,i.mu,i.m2 = 0,0,0
  elseif i.n > 0 then
    d = v-i.mu
    i.mu = i.mu + w*d/i.n
    i.m2 = i.m2 + w*d*(v-i.mu) end end

Welford's online update, one assignment per term. Missing values ("?") skip. The guard handles the edge case where a subtraction would push n below one; the column resets instead of going unstable. Subtraction is as cheap as addition because Welford is symmetric under negation. The active learner in Chapter 5 uses this to move rows between sets without rebuilding anything.

Adding a row threads method dispatch through three layers of metatables:

function DATA._add(i,row,w)
  if not i.cols then i.cols = Cols(row)
  else i._mid = nil
       add(i.cols,row,w)
       if w>0 then l.push(i.rows,row)
       else
         for n,r in ipairs(i.rows) do
           if r==row then
             table.remove(i.rows,n); break end end end end end

function COLS._add(i,row,w)
  for _,c in ipairs(i.all) do add(c,row[c.at],w) end
  return row end

One call on the Data recurses into its Cols, which maps across every column, dispatching to either the Num or Sym method via its metatable. No isinstance checks at any call site. The metatable knows.

Fifty lines of constructors and _add methods give us a running summary of any CSV, row removal as cheap as row addition, role assignment from column names alone, and a type system that dispatches for us. No DataFrame library, no ORM, not even Lua's own optional type-checkers. The data carries its own schema.

Try it yourself

Write --csv: read a CSV into a list of rows. Each row is a list of values. Strings that parse as numbers become numbers. "?" becomes a distinct missing marker. Everything else stays a string.

Write --data: take that list. Treat the first row as header. Build a Num or Sym per column from its first character. Assign to features or goals from its last character. Feed remaining rows through so each column updates. Print the goal columns.

Combined, reading and summarizing a CSV online is maybe forty lines of Lua. None of it requires a library.

3. Online stats

Given a column of four hundred numeric values, how do you answer "what is typical" and "how spread out is it" without storing all four hundred values in a list?

Plot a column on a histogram. A curve emerges. Peaks where rows cluster, thin tails where they do not. That curve is a probability density function, or PDF. The cumulative form, the fraction of the column at or below x, is the CDF.

For most real-world columns the PDF is roughly a bell. A Gaussian. Every Gaussian is two numbers: a center mu and a width sigma. Summarize those and you have the distribution.

A naive mu sums every value and divides by count. That needs either all the values in memory or two passes over the file. A naive sigma sums squared deviations from mu, which needs a first pass to get mu and a second for the deviations. Welford solved both in 1962. Keep mu and a helper m2, the running sum of squared deviations from the current mu. Update in O(1) per value. After n values, sigma = sqrt(m2/(n-1)).

The three arithmetic lines, already shown in Chapter 2:

i.n = i.n + w
d = v - i.mu
i.mu = i.mu + w*d/i.n
i.m2 = i.m2 + w*d*(v - i.mu)

One assignment per sample. No list. mu is always current. Anything derived from mu and m2 costs O(1) to read. Pass w=-1 and the same formulas unwind a previous add. Rows can be discarded after they pass through; the Num remembers what matters. This is single-source-of-truth for the column. The summary is not built from stored rows on demand. The summary is the stored form.

The concept of a center and a width exists for every column type, though the math differs. For a Num, center is mu and spread is sigma. For a Sym, center is the most frequent value (the mode) and spread is an entropy-style measure over the frequency dictionary. For a Data, center is the centroid: a vector of per-column centers, memoized until the next add.

function NUM.mid(i) return i.mu end
function SYM.mid(i,    most,mode)
  most = -1
  for v,n in pairs(i.has) do
    if n>most then most,mode = n,v end end
  return mode end

function NUM.spread(i)
  return i.n>1 and (max(0,i.m2)/(i.n-1))^0.5 or 0 end
function SYM.spread(i)
  return -l.sum(i.has,
    function(v) return v/i.n*log(v/i.n,2) end) end

The caller never writes if getmetatable(col) == NUM. That check lives once, at method dispatch, as the metatable __index walk. Add a new column type (a date column, say) by writing new methods on a new metatable. No caller edits. This is what open for extension, closed for modification buys you.

The last piece is normalization. Two columns often have different scales. Mpg lives in 10 to 40. Cylinders lives in 3 to 8. You cannot mix their deviations; the Mpg term would swamp the Cylinders one. Normalize each to a common [0, 1]:

function NUM.norm(i,v,    z)
  if v=="?" then return v end
  z = (v - i.mu)/(i:spread() + 1e-32)
  return 1/(1 + exp(-1.7*l.crop(z,-3,3))) end

Compute z, the z-score. Clamp to [-3, 3], which covers 99.7 percent of a Gaussian. Apply a sigmoid at scale 1.7, a known approximation to the Gaussian CDF. The output is the approximate fraction of the column at or below v. Every column's values now live on a common [0, 1], comparable. For a Sym, norm is the identity. Equality is the only comparison that matters.

Chapter 4 wants to compute distances between rows. That works across columns of different types and scales only if every value reduces to a number in [0, 1] in O(1). Welford gave us mu and m2 in O(1). m2 gives us sigma in O(1). sigma gives us z. The sigmoid gives us the CDF approximation. Each layer constant-time because the layer underneath it was.

Try it yourself

Write --stats: walk every column in a loaded Data and print its name, mid, and spread. The key test is that you never iterate rows inside this command. If you find yourself computing the mean from a list, Chapter 2 went wrong.

Write --norm: pick the first row. For each goal column, print the raw value and the normalized value. Numeric columns should map into (0, 1). Symbolic columns should pass through. Verify that norm(col, mid(col)) is close to 0.5: the center of the CDF.

In Lua, maybe ten lines on top of Chapter 2. No stats library required.

4. Distance

Chapter 3 gave us one number per column on a common [0, 1] scale. Chapter 4 turns per-column numbers into per-row numbers. Distance.

Two distances do two jobs. disty measures how far a row is from perfection — the "heaven" row where every minimize-column is at its minimum and every maximize-column is at its maximum. distx measures how similar two rows are on features alone. disty uses only y-columns; distx uses only x-columns. They never cross.

disty plugs in "how far is this value from heaven":

function DATA.disty(i,row,    fn)
  fn = function(c) return
    abs(c:norm(row[c.at]) - c.heaven)^the.p end
  return (l.sum(i.cols.y,fn)/#i.cols.y)^(1/the.p) end

Normalize the value to [0, 1]. Subtract heaven (0 or 1). Raise to the.p (the Minkowski exponent). Sum across y-columns, divide by count, take the p-th root. One scalar per row, lower is better. With p=1 this is Manhattan. With p=2 it is Euclidean.

distx delegates per column, because different column types need different distance math:

function NUM.aha(i,a,b)
  if a=="?" and b=="?" then return 1 end
  a,b = i:norm(a), i:norm(b)
  if a=="?" then a = b>0.5 and 0 or 1 end
  if b=="?" then b = a>0.5 and 0 or 1 end
  return abs(a-b) end

function SYM.aha(i,a,b)
  if a=="?" and b=="?" then return 1 end
  return a==b and 0 or 1 end

function DATA.distx(i,r1,r2,    fn)
  fn = function(c) return c:aha(r1[c.at],r2[c.at])^the.p end
  return (l.sum(i.cols.x,fn)/#i.cols.x)^(1/the.p) end

Two methods of aha. The sym method is equality: zero if match, one if not. The num method normalizes first, then takes absolute difference. When one side is missing, assume the far end of the scale from whatever the known side is. Missing as far as possible from anything that is not missing.

Twenty lines, pairwise distance on mixed-type rows with missing-value handling. A Python equivalent using NumPy plus pandas plus scikit-learn's NearestNeighbors plus an imputation library would be orders of magnitude more code, across several libraries, opaque to audit.

Notice what the metatable buys you. If tomorrow we add a Date column type, aha needs one new method on a new metatable and every existing caller works unchanged. This is open-closed doing real work.

Try it yourself

Write --ydata: build a Data, sort its rows by disty, print every thirtieth. You will see the best rows clustered at the top. Write --xdata: take the first row as a reference, sort every other row by distx to it, print every thirtieth. You will see the neighbors. The two commands together show that y and x are two separate organizations of the same rows. Chapter 5 exploits that.

5. Active learning

Labels cost money. Reading an already-written row is free; writing the ground-truth y-value for it is not. In software engineering an expert annotation takes minutes to hours. A fuzzer run that confirms a bug takes compute. A log review takes attention. Four hundred rows to label is a two-day task.

So pick carefully. Do not label everything. Do not label randomly. Label the rows where labeling will teach you something.

The classical approach is probabilistic: estimate the learner's uncertainty on each unlabeled row, pay for the most uncertain one. Correct, but heavyweight, because it needs a probabilistic model. Here we use a filter that needs two distx calls and a less-than.

The trick: keep two labeled subsets. best holds the good-looking rows; rest holds the rest. Both are real Data objects because we want their running statistics, and thus their centroids, to be current.

The filter: is this new row closer to best's centroid or to rest's?

function closer(lab,best,rest,r)
  return lab:distx(r,best:mid()) < lab:distx(r,rest:mid()) end

If closer to best, the row looks promising. Spend a label. If not, skip. Two distx calls. That is the whole acquisition heuristic.

A warm start seeds the centroids. Four random labels, sorted by disty, split top-sqrt from the rest. Then the main loop scans a capped pool of unlabelled rows and admits any that pass the filter, up to budget:

function active(rs,data,    lab,best,rest,ys,sorted,n)
  lab, best, rest = data:clone(), data:clone(), data:clone()
  ys = Num()
  for k = 1, min(4,#rs) do
    add(lab,rs[k]); add(ys, lab:disty(rs[k])) end
  sorted = {}
  for _,r in ipairs(lab.rows) do l.push(sorted,r) end
  l.sort(sorted,
    function(a,b) return lab:disty(a) < lab:disty(b) end)
  n = max(1, floor((#sorted)^0.5))
  for k = 1, n         do add(best, sorted[k]) end
  for k = n+1, #sorted do add(rest, sorted[k]) end
  for k = 5, min(the.few, #rs) do
    if ys.n >= the.Budget then break end
    if closer(lab,best,rest,rs[k]) then
      add(ys, lab:disty(rs[k]))
      add(lab, rs[k]); add(best, rs[k])
      rebalance(best,rest,lab) end end
  return best, lab, ys.n end

rebalance caps best at sqrt(|lab|), evicting its worst-disty row into rest. Keep best small. Keep lab as the running ledger of paid-for labels.

The whole loop is twenty lines. No entropy, no uncertainty, no probabilistic model. The filter is "does this row look like the ones we liked?" and the answer comes from two distx calls.

Now the evaluation question. Does it actually work?

Honest evaluation needs a holdout. Split rows 50/50 into train and test. Run active on train. Build a tree from what it labeled (next chapter). Rank test rows by the tree's predictions, using only their features. Pay for the top Check ground-truth labels. Pick the winner. Score that winner against a god-view of the full dataset:

function validate(rows,data,win)
  l.shuffle(rows)
  local n = #rows // 2
  local train = l.slice(rows, 1, n)
  local test  = l.slice(rows, n+1)
  local _, lab, lbl = active(train, data)
  local tb, tbD = nil, 1E32
  for _,r in ipairs(lab.rows) do
    local d = data:disty(r)
    if d<tbD then tb, tbD = r, d end end
  local tr = Tree(function(r) return lab:disty(r) end)
                :build(lab, lab.rows)
  l.sort(test,
    function(a,b) return tr:leaf(a).y:mid() < tr:leaf(b).y:mid() end)
  local top = l.sort(l.slice(test,1,the.Check),
    function(a,b) return data:disty(a) < data:disty(b) end)
  return win(tb), win(top[1]), lbl + the.Check end

Every god-disty call is counted. Total label cost per run is Budget + Check: 50 + 5 = 55 on defaults. No y-leaks anywhere during training; the tree never sees a god-disty, only the labels active paid for.

The result on auto93.csv: median hold-win around 94 out of 100 at Budget=50. For comparison: label 55 rows at random and pick the best gets around 50. A full-dataset baseline with access to every label tops out around 97. Fifty-five labels get us to 97 percent of full-dataset performance.

Try it yourself

Write --active: run validate twenty times with different seeds, print each run's train-win, hold-win, and labels-used. Sort by hold-win and take the median. At Budget=50 it should land near 94. At Budget=30, with centroid ranking, around 91. With tree ranking, around 85. The six-point gap at thirty labels is the explanation tax, which Chapter 6 is about.

6. Trees and the explanation tax

Chapter 5 ends with "build a tree, rank holdout by leaf-predicted y-value." This chapter builds that tree and counts what it costs.

A decision tree partitions rows into buckets. At each internal node, one column and one cut value. Rows with column-value below the cut go left, above go right. Each leaf is a bucket; its predicted y is the mean of its rows' disty values.

With the stats of Chapter 3 and the distance of Chapter 4 in hand, most of the tree is composition.

Choosing cuts on numeric columns is where traditional CART gets expensive and overfits. Standard CART sorts the values and evaluates every candidate threshold, picking whichever minimizes training impurity. With fifty labels that means forty-nine cut candidates per numeric column, and the "best" one is often the one that happened to separate a noise-value from the rest. The optimizer finds spurious structure in sparse data.

We regularize by fiat. For numeric columns, only consider the median:

function NUM.splits(i,rows,fn,    vs,mu,sp)
  vs = {}
  for _,r in ipairs(rows) do
    if r[i.at]~="?" then l.push(vs,r[i.at]) end end
  if #vs<2 then return {} end
  l.sort(vs)
  mu = vs[#vs//2+1]
  sp = split(i,rows,fn,mu,function(v) return v<=mu end)
  return sp and {sp} or {} end

One cut per numeric column. A strong prior: we believe the median is good enough regardless of the training labels. Trades theoretical optimality for variance reduction. Works because the trees stay shallow and are only asked to rank rough regions, not make precise predictions.

Greedy split selection over all (column, cut) pairs picks whichever minimizes weighted spread of its children:

function split(col,rows,fn,cut,test,    lhs,rhs,L,R,ok)
  lhs,rhs,L,R = Num(),Num(),{},{}
  for _,r in ipairs(rows) do
    ok = r[col.at]=="?" or test(r[col.at])
    l.push(ok and L or R, r); add(ok and lhs or rhs, fn(r)) end
  if #L>=the.leaf and #R>=the.leaf then
    return {col=col,cut=cut,left=L,right=R,lhs=lhs,rhs=rhs} end end

The same spread method handles both regression and classification because Chapter 3's polymorphism already resolved the type. Same impurity code for both tasks.

Recurse until each node has fewer than 2*leaf rows. Default leaf=3, so nodes stop splitting at five rows. Print with mid on the left, size in parens, condition on the right:

0.35 ( 27)   Mpg <= 25
0.30 ( 13)   |   Hp <= 100
0.41 ( 14)   Mpg >  25

Three columns of information: node mean disty, node size, branch condition. Read top to bottom and every decision the tree made is visible. Try that with a transformer.

This is the XAI payoff. Rudin called for it2: high-stakes decisions deserve interpretable models, not post-hoc explanations of opaque ones. Here is an interpretable model, in eighty lines.

But it is not free. We measured the tax. On auto93 at Budget=30, ranking the holdout via a tree costs six points of median hold-win versus ranking via a single best centroid. At Budget=50 the tax is zero. Above fifty labels the tree actually wins.

The reason is variance. Each leaf's mean disty comes from a handful of rows. With few labels, those means are noisy and the tree routes test rows wrongly. A centroid is a single stable point: less expressive, also less variable. At small N, variance dominates bias. At large N, bias dominates. Trees win when there is enough data to stabilize the leaf estimates.

The takeaway for interpretability advocates: your model has a minimum sample size. Below it, you are paying an explanation tax on noise-amplified decision regions. Above it, the tax vanishes. Build both rankers. Show both numbers. Let the reader pick.

Try it yourself

Write --tree: pick a budget of random rows, grow a tree from them, print it. Try leaf=3, leaf=5, leaf=7. Smaller leaves give more decision regions and more variance. Larger leaves give fewer regions and more stability. Find your sweet spot for a dataset you care about.

7. Notation

Why Lua? Not for multi-dispatch, because Lua does not have it. But Lua has something better for this code: metatables.

A metatable is a plain table that tells another table how to behave. Attach SYM as the metatable of a sym-shaped table, and i:_add(v,w) looks up _add on SYM, finds the right function, calls it with i as first argument. This gives us Julia-like first-argument dispatch in ten lines:

function l.new(kl,obj) kl.__index = kl; return setmetatable(obj,kl) end

That is our entire type system. Num, Sym, Cols, Data, Tree are five empty tables. Objects are tables with one of those as their metatable. Method calls walk the metatable chain. No classes, no inheritance, no type declarations.

For this code, we only ever dispatch on the first argument (always Num/Sym/Data/Tree). Single-dispatch via metatable gives us that cheaply. Julia's full multiple-dispatch machinery would dispatch on any argument combination; we do not use that capacity. Lua's simpler model is enough.

Compared numerically across four implementations of the same pipeline:

file       lines   code-bytes   hold-win   runtime
fri.lisp    495        15,172      93.5    0.111 s
ezr.jl      387         9,632      89.5    2.948 s
ezr.lua     393        11,351      94.5    0.058 s
ezr.py      543        19,423      84      0.082 s

Julia is fewest lines and fewest code-bytes. Lua is fastest runtime and highest hold-win. Python lags on both size and speed. Lisp sits in the middle.

What Lua gives you that the others do not:

Tiny startup. A Lua script runs in ten milliseconds. Lisp via SBCL ships a pre-compiled image but still pays fifty milliseconds of init. Julia JIT-compiles on first call, taking seconds. Python's interpreter pays thirty milliseconds. For a bounded workload like twenty 400-row iterations, startup dominates. Lua wins because Lua has less to start.

One data structure. Tables do everything: dicts, arrays, objects, namespaces. No dict-vs-list-vs-class taxonomy. The library code in Chapter 2 works because Lua's table is flexible enough to be both the frequency map in a Sym and the row list in a Data.

String pattern matching as a built-in. s:match"^[A-Z]" is one call. Python has re, Julia has match with r"...", Lisp needs cl-ppcre. Lua's patterns are not full regex (no alternation, limited character classes), but they cover ninety percent of parse-tasks in thirty percent of the characters.

What Lua costs you: no multiple dispatch on several argument types, no keyword arguments, no default-function-parameter system beyond and/or idioms (x = x or default). For this code none of those are load-bearing.

You can port ezr.lua in a weekend. The exercise list at the end of each chapter is the port guide. Pick your target. The numbers will be close; the shape will differ.

Try it yourself

Pick a language and port one function. NUM.aha is a good first choice: seven lines here, testable against the formula directly. Count the bytes of your port. That is the cost of the language on this kind of code. Usually within thirty percent either direction of Lua.

8. What we rejected

Along the way we declined two practices most engineers would default to. Both choices matter for the argument.

First, we did not write unit tests. No test framework, no CI of the usual kind. The file contains a handful of runnable --* demos. --the prints the config. --csv reads and prints a file. --data builds a Data. And so on up to --active, which runs twenty evaluation splits and prints a results table.

Why no unit tests? Because correctness at the unit level is not well-defined for most of this code. What is the correct output of TREE.build on a random five-row subset? The only meaningful test is statistical: across twenty runs at a known seed, does the median hold-win stay near ninety-four? We keep reference implementations — fri.lisp in Common Lisp, ezr.jl in Julia, ezr.py in Python — that reach the same number. Refactors are validated by running --active on fixed seeds and comparing medians against all three.

This is test-driven development's statistical cousin. The reference implementation is the oracle. The running numbers are the passing bar. It works here because the subject is a learner, not a business rule. If we were validating a billing engine we would have unit tests. Rule of thumb: unit tests for local correctness, statistical evaluation for emergent correctness, pick the right tool for the subject.

Second, we did not document everything. The file has one-line comments on the non-obvious functions, not on the obvious ones. closer, rebalance, warm_start, active, validate, wins, TREE.build, NUM.splits, the aha methods, the library primitives: all documented with a single line. mid, spread, add, sub, every _add method, every eg["--*"] entry point: no doc at all. Names are enough.

Why the silence? Because comments on obvious functions decay into noise. Three months after writing -- Adds v to the column summary on add, the comment is useless. New readers learn to skip comments entirely, which means the comments that actually matter — the ones on non-obvious functions — also get skipped. You lose the leverage of the useful comment by paying for the useless one.

The top-of-file help block, by contrast, earns its place. It documents the CLI commands, the option flags with defaults, the CSV format, the naming conventions, and the config-parsing regex right below. Forty lines of documentation that pay for themselves every time someone opens the file. The same help block is also where the config lives — the parser extracts defaults from the same text a reader uses for reference.

Both choices are instances of the same principle. Do what pays. Skip what does not. Unit tests and document-everything are tools, not rules. Deploy where they earn their place.

Try it yourself

List three practices you consider mandatory in your daily work. For each, name what the practice is paying for, not the name of the principle it claims. Decide whether, for the current project, that payment is worth it. Keep the ones that pay. Drop the ones that do not. Treat the list as a working document, not a creed.

9. Results

On auto93.csv, a 398-row dataset with eight features and three goal columns (Mpg-, Acc+, Weight-), the ezr.lua pipeline:

lua ezr.lua -s 1 -B 50 --active auto93.csv

runs in 58 milliseconds and produces twenty evaluation rows. Median train-win: 97. Median hold-win: 94.5 at seed=1, 91.1 averaged across five seeds. Label cost per run: 55 (50 from active, 5 from Check).

A random baseline (label 55 rows at random, keep the best) gets median hold-win around 50. A full-dataset baseline with access to every label tops out around 97. Fifty-five labels get us to 97 percent of full-dataset performance.

The tree-ranked variant matches the centroid baseline at Budget=50 and loses six points at Budget=30. The explanation tax is a threshold, not a slope.

Four implementations of the same pipeline, four measurements each:

file       lines   code-bytes   hold-win   runtime
ezr.lua     393        11,351      94.5    0.058 s  <-- this book
ezr.jl      387         9,632      89.5    2.948 s
fri.lisp    495        15,172      93.5    0.111 s
ezr.py      543        19,423      84      0.082 s

Julia wins fewest lines and fewest code-bytes. Lua wins highest hold-win and fastest runtime. Julia has fewest code-bytes; its JIT warmup costs it on a per-invocation benchmark but would pay off at longer workloads. Lisp sits in the middle on every axis. Python lags on both size and accuracy.

Zero external dependencies in any of the four beyond the respective runtime. All reproducible under a seed.

You can read any of them top to bottom in twenty minutes. You can re-implement them in a weekend. The Try it yourself section at the end of every chapter is the porting guide.

Closing

Knuth chose to be on the bottom of things. In an age of LLMs generating code faster than anyone can verify it, someone has to. Read the code. Compile it. Run the numbers. Port it. Own the whole chain. If you cannot see the whole algorithm on one screen, you do not understand it. You trust it. Trust is a bug.

Try it yourself

Clone ezr.lua. Run lua ezr.lua -s 1 -B 50 --active auto93.csv. Confirm the median. Then start your port. When you finish, come back and compare.

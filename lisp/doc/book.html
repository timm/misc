1. Motivation

Email is a wonderful thing for people whose role in life is to be on top of things. But not for me; my role is to be on the bottom of things.
— Donald Knuth1

In an age where everyone hands details to an LLM, what are we missing?

We refactored an AI research codebase to the bone. Four hundred lines of Julia, one file, no dependencies beyond Julia's standard library, runs in milliseconds once the JIT has warmed. Five things stopped being true.

One. Regression, classification, and optimization are the same task. Our active-learning loop does all three. The only difference is a suffix in the CSV header. A column named Mpg- is a minimization target. Mpg+ is maximization. class! is a classifier label. Everything else is a feature. Three textbook chapters, one algorithm.

Two. Five abstractions, everywhere. Num, Sym, Cols, Data, and a distance function. Every module composes from these. The Num that tracks mean and standard deviation is three lines of slots. That same Num is the beating heart of the active learner, the splitting criterion for decision trees, and the basis of every "is X better than Y" judgement the code makes. Build once, reuse always.

Three. Interpretability is cheap. Rudin argued that high-stakes decisions should use interpretable models, not explained black boxes2. We agree. The cost is small. Our decision tree learner is sixty lines of Julia. Trees print on a page. We measured the tax: at thirty labels, a tree-ranked holdout loses six points of win to a centroid baseline. At fifty labels the tax is zero. Interpretability has a minimum sample size. Above it, it costs nothing.

Four. Fewer than fifty labels beat big data. Labels in software engineering are expensive. Expert annotation fatigues quickly. Historical logs are unreliable; up to ninety percent of technical-debt entries are false positives3. Automated oracles use approximate heuristics that miss the interesting cases. Active learning picks informative rows and stops. Fifty labels outperform baselines trained on thousands.

Five. Dependencies are a supply-chain bet. Modern software is eighty-five to ninety-seven percent external code4. Each dependency widens the attack surface. A single poisoned npm or PyPI update compromises thousands of downstream packages silently. Richard Hipp, author of SQLite, practices backpacking: cutting code to what is essential. He built his own editor, servers, storage engine. He shunned Berkeley DB. Oracle later acquired it and demanded license fees. SQLite ships today on every Android phone, every iPhone, Chrome, Firefox, Safari, most Linux distributions, and Mars rovers. Hipp's framing: if you want to be free, that means doing things yourself5. Our CSV reader is eight lines. Our CLI parser is twelve. The whole file is four hundred.

Why Julia?

Julia is the mainstream language closest to Lisp in the ways that matter for AI-math code. It has real multiple dispatch as first-class syntax — the single feature most responsible for this file being four hundred lines instead of fifteen hundred. It has keyword arguments, list comprehensions, and @kwdef structs with defaults. It JITs to near-C speed after the first call. It reads Python-ish, so any ML researcher can follow it without learning a dialect.

You may not know Julia. That is fine. The code is short, the shape carries. function add!(n::Num, v, w=1) ... end defines a method dispatched on the Num type. struct, for, return look like what they look like everywhere. Learn the rest as you go. Every chapter ends with a Try it yourself list of the exercises a port would need in your language of choice.

The claim

If you cannot see the whole algorithm on one screen, you do not understand it. You trust it. Trust is a bug.

Try it yourself

Print the config. In ezr.jl it lives in a list called THE. Each entry is a 4-tuple (name, default, flag, description). The CLI parser walks ARGS, matches flags, overwrites defaults via the the dict. No argparse, no dictionary of globals. One list that is both help text and storage.

Implement this in your language. A list of 4-tuples, a flag parser that finds an entry by its third field, a CLI loop that overwrites the second field. The description field doubles as --help. The point is feeling how config-as-data cuts out a library.

2. The landscape

The field is called data mining. Most textbooks ignore the data.

They race to algorithms. Decision trees, gradient descent, backprop, transformers. Each one works only because the input has regularities: correlations, clusters, separable classes, distributions that repeat themselves. If rows were random noise, no algorithm would help. So what shape is the data?

Two regularities matter. First, values have types. Some are numbers; some are words. You cannot add "five" and "three" but you can add 5.0 and 3.0. You cannot ask for the mean of ["red", "green", "blue"], but the mode is fine. Two struct types cover it:

@kwdef mutable struct Sym
  at::Int=0; txt::String=" "; n::Int=0; has=Dict() end

@kwdef mutable struct Num
  at::Int=0; txt::String=" "; n::Int=0
  mu::Float64=0.0; m2::Float64=0.0; heaven::Int=1 end

Both carry a position (at), a name (txt), a count (n). A Sym keeps a frequency dictionary. A Num keeps Welford's running mean (mu) and sum of squared deviations (m2). heaven is the direction of good: 1 for maximize, 0 for minimize. Keep types separate and no downstream code has to branch on type.

Second regularity: names carry roles. Capitalize numerics. Suffix goals with + or -. Mark a classifier target with !. Mark skipped columns with X. The classifier is eight lines:

function make_cols(names)
  all = [make_col(i, t) for (i, t) in enumerate(names)]
  x, y = Any[], Any[]
  for c in all
    endswith(c.txt, "X") && continue
    occursin(r"[!+\-]$", c.txt) ?
      push!(y, c) : push!(x, c) end
  Cols(names, all, x, y) end

Capital first letter gives a Num. Lowercase gives a Sym. The last character sets the role. Convention over configuration: the data tells us what the data is. No registry, no manifest, no plugin file.

Data is rows plus their summary:

@kwdef mutable struct Data
  rows::Vector=Any[]; cols=nothing; mid=nothing end

One idea drives the design: the stats are the data. There is no DataFrame of statistics alongside a list of rows. Every column object carries its own running summary. Ask for the mean; read mu. No separate fit step. Pop a row with sub! and the summaries correct themselves.

The mutation vocabulary has four verbs. add! creates, sub! deletes, field-access reads (n.mu), assignment updates (n.mu = x). CRUD, the same four every REST API pretends to have. add! pushes a value into a column's summary and returns the value:

function add!(n::Num, v, w=1)
  unk(v) && return v
  if w < 0 && n.n <= 2
    n.n = 0; n.mu = 0.0; n.m2 = 0.0
  else
    n.n += w
    d = v - n.mu
    n.mu += w * d / n.n
    n.m2 += w * d * (v - n.mu) end
  v end

Welford's online update, one assignment per term. Missing values ("?") skip. The guard handles the edge case where a subtraction would push n below two; the column resets instead of going unstable. sub! is add! with w = -1. Subtraction is as cheap as addition because Welford is symmetric under negation. The active learner in Chapter 5 uses this to move rows between sets without rebuilding anything.

Adding a row threads multiple dispatch through three layers:

function add!(d::Data, row, w=1)
  d.mid = nothing
  if d.cols === nothing
    d.cols = make_cols(row)
  else
    add!(d.cols, row, w)
    w > 0 ? push!(d.rows, row) :
            deleteat!(d.rows, findfirst(==(row), d.rows)) end
  row end

function add!(cs::Cols, row, w=1)
  for (c, v) in zip(cs.all, row); add!(c, v, w); end
  row end

One call on the Data recurses into its Cols, which maps across every column, dispatching to either the Num or Sym method. No isinstance checks at any call site. The method machinery knows.

Sixty lines of struct and multiple-dispatch methods give us a running summary of any CSV, row removal as cheap as row addition, role assignment from column names alone, and a type system that dispatches for us. No DataFrames.jl, no StatsBase, no ORM. The data carries its own schema.

Try it yourself

Write --rows: read a CSV into a list of rows. Each row is a list of values. Strings that parse as numbers become numbers. "?" becomes a distinct missing marker. Everything else stays a string. In Python, about fifteen lines; in Julia, about eight.

Write --data: take that list. Treat the first row as header. Build a Num or Sym per column from its first character. Assign to features or goals from its last character. Feed remaining rows through so each column updates. Print the goal columns. In Python, about thirty lines; in Julia, about twenty.

Combined, reading and summarizing a CSV online is maybe fifty lines of Julia. None of it requires a library beyond Printf.

3. Online stats

Given a column of four hundred numeric values, how do you answer "what is typical" and "how spread out is it" without storing all four hundred values in a list?

Plot a column on a histogram. A curve emerges. Peaks where rows cluster, thin tails where they do not. That curve is a probability density function, or PDF. The cumulative form, the fraction of the column at or below x, is the CDF.

For most real-world columns the PDF is roughly a bell. A Gaussian. Every Gaussian is two numbers: a center mu and a width sigma. Summarize those and you have the distribution.

A naive mu sums every value and divides by count. That needs either all the values in memory or two passes over the file. A naive sigma sums squared deviations from mu, which needs a first pass to get mu and a second for the deviations. Welford solved both in 1962. Keep mu and a helper m2, the running sum of squared deviations from the current mu. Update in O(1) per value. After n values, sigma = sqrt(m2 / (n - 1)).

The three arithmetic lines, already shown in Chapter 2:

n.n += w
d = v - n.mu
n.mu += w * d / n.n
n.m2 += w * d * (v - n.mu)

One assignment per sample. No list. mu is always current. Anything derived from mu and m2 costs O(1) to read. Pass w = -1 and the same formulas unwind a previous add. Rows can be discarded after they pass through; the Num remembers what matters. This is single-source-of-truth for the column. The summary is not built from stored rows on demand. The summary is the stored form.

The concept of a center and a width exists for every column type, though the math differs. For a Num, center is mu and spread is sigma. For a Sym, center is the most frequent value (the mode) and spread is an entropy-style measure over the frequency dictionary. For a Data, center is the centroid: a vector of per-column centers, memoized until the next add!.

mid(n::Num) = n.mu
mid(s::Sym) = argmax(s.has)
function mid(d::Data)
  d.mid === nothing &&
    (d.mid = [mid(c) for c in d.cols.all])
  d.mid end

spread(n::Num) = n.n < 2 ? 0.0 :
  sqrt(max(0.0, n.m2) / (n.n - 1))
spread(s::Sym) = -sum(let p = v / s.n; p * log2(p) end
  for v in values(s.has) if v > 0; init=0.0)

The caller never writes if typeof(col) == Num. That branch lives once, at method dispatch. Add a new column type (a date column, say) by writing new methods. No caller edits. This is what open for extension, closed for modification buys you. The if-ladder does not grow by a branch.

The last piece is normalization. Two columns often have different scales. Mpg lives in 10 to 40. Cylinders lives in 3 to 8. You cannot mix their deviations; the Mpg term would swamp the Cylinders one. Normalize each to a common [0, 1]:

norm(::Sym, v) = v
function norm(n::Num, v)
  unk(v) && return v
  z = (v - n.mu) / (spread(n) + 1e-32)
  1 / (1 + exp(-1.7 * clamp(z, -3, 3))) end

Compute z, the z-score. Clamp to [-3, 3], which covers 99.7 percent of a Gaussian. Apply a sigmoid at scale 1.7, a known approximation to the Gaussian CDF. The output is the approximate fraction of the column at or below v. Every column's values now live on a common [0, 1], comparable. For a Sym, norm is the identity. Equality is the only comparison that matters.

Chapter 4 wants to compute distances between rows. That works across columns of different types and scales only if every value reduces to a number in [0, 1] in O(1). Welford gave us mu and m2 in O(1). m2 gives us sigma in O(1). sigma gives us z. The sigmoid gives us the CDF approximation. Each layer constant-time because the layer underneath it was.

Try it yourself

Write --stats: walk every column in a loaded Data and print its name, mid, and spread. The key test is that you never iterate rows inside this command. If you find yourself computing the mean from a list, Chapter 2 went wrong.

Write --norm: pick the first row. For each goal column, print the raw value and the normalized value. Numeric columns should map into (0, 1). Symbolic columns should pass through. Verify that norm(col, mid(col)) is close to 0.5: the center of the CDF.

In Python, maybe twenty lines on top of Chapter 2. In Julia, about ten. No stats library required.

4. Distance

Chapter 3 gave us one number per column on a common [0, 1] scale. Chapter 4 turns per-column numbers into per-row numbers. Distance.

Two distances do two jobs. disty measures how far a row is from perfection — the "heaven" row where every minimize-column is at its minimum and every maximize-column is at its maximum. distx measures how similar two rows are on features alone. disty uses only y-columns; distx uses only x-columns. They never cross.

Both compose from the same Minkowski kernel:

function minkowski(vs)
  p = the[:p]; tot = 0.0; k = 0
  for v in vs; tot += v^p; k += 1; end
  (tot / max(1, k))^(1 / p) end

With p=1 this is Manhattan distance. With p=2 it is Euclidean. Pick on the CLI with -p. The kernel does not know about columns. It takes an iterator of per-element distances and combines them. All per-column math lives outside.

disty builds the iterator "how far is this value from heaven":

disty(d, row) = minkowski(
  abs(norm(c, cell(c, row)) - c.heaven) for c in d.cols.y)

Normalize the value to [0, 1]. Subtract heaven (0 or 1). Take absolute value. Minkowski-combine across y-columns. One scalar per row, lower is better.

distx delegates per column, because different column types need different distance math:

function aha(::Sym, a, b)
  unk(a) && unk(b) && return 1.0
  a == b ? 0.0 : 1.0 end

function aha(c::Num, a, b)
  unk(a) && unk(b) && return 1.0
  a, b = norm(c, a), norm(c, b)
  unk(a) && (a = b > 0.5 ? 0.0 : 1.0)
  unk(b) && (b = a > 0.5 ? 0.0 : 1.0)
  abs(a - b) end

distx(d, r1, r2) = minkowski(
  aha(c, cell(c, r1), cell(c, r2)) for c in d.cols.x)

Two methods of aha. The sym method is equality: zero if match, one if not. The num method normalizes first, then takes absolute difference. When one side is missing, assume the far end of the scale from whatever the known side is. Missing as far as possible from anything that is not missing.

Twenty lines, and you have pairwise distance on mixed-type rows with missing-value handling. A Python equivalent using NumPy plus pandas plus scikit-learn's NearestNeighbors plus an imputation library would be orders of magnitude more code, across several libraries, opaque to audit.

Notice what multiple dispatch buys you. If tomorrow we add a Date column type, aha needs one new method and every existing caller works unchanged. This is open-closed doing real work.

Try it yourself

Write --ydata: build a Data, sort its rows by disty, print every thirtieth. You will see the best rows clustered at the top. Write --xdata: take the first row as a reference, sort every other row by distx to it, print every thirtieth. You will see the neighbors. The two commands together show that y and x are two separate organizations of the same rows. Chapter 5 exploits that.

5. Active learning

Labels cost money. Reading an already-written row is free; writing the ground-truth y-value for it is not. In software engineering an expert annotation takes minutes to hours. A fuzzer run that confirms a bug takes compute. A log review takes attention. Four hundred rows to label is a two-day task.

So pick carefully. Do not label everything. Do not label randomly. Label the rows where labeling will teach you something.

The classical approach is probabilistic: estimate the learner's uncertainty on each unlabeled row, pay for the most uncertain one. Correct, but heavyweight, because it needs a probabilistic model. Here we use a filter that needs two distx calls and a less-than.

The trick: keep two labeled subsets. best holds the good-looking rows; rest holds the rest. Both are real Data objects because we want their running statistics, and thus their centroids, to be current.

The filter: is this new row closer to best's centroid or to rest's?

closer(lab, best, rest, r) =
  distx(lab, r, mid(best)) < distx(lab, r, mid(rest))

If closer to best, the row looks promising. Spend a label. If not, skip.

A warm start seeds the centroids. Four random labels, sorted by disty, split top-sqrt from the rest:

function warm_start!(rs, lab, best, rest, ys)
  for _ in 1:min(4, length(rs))
    r = popfirst!(rs)
    add!(lab, r); add!(ys, disty(lab, r)) end
  sorted = sort(lab.rows, by = r -> disty(lab, r))
  n = max(1, Int(floor(sqrt(length(sorted)))))
  for r in sorted[1:n];     add!(best, r); end
  for r in sorted[n+1:end]; add!(rest, r); end
  rs end

Then the main loop. Cap the remaining pool at the[:u] rows. Apply the filter to each row in turn. When closer, label it and add it to both lab (the record of everything labeled) and best (promising subset). Optionally rebalance.

function active(rs)
  hd   = first(rs)
  lab  = make_data([hd])
  best = make_data([hd])
  rest = make_data([hd])
  ys   = Num()
  pool = warm_start!(
    shuffled(rs[2:end]), lab, best, rest, ys)
  for r in pool[1:min(the[:u], length(pool))]
    ys.n >= the[:b] && break
    if closer(lab, best, rest, r)
      add!(ys, disty(lab, r))
      add!(lab, r); add!(best, r)
      rebalance!(best, rest, lab) end end
  best, lab, ys.n end

Rebalance: when best grows past sqrt(|lab|), evict its worst-disty row into rest. Keep best small. Keep lab as the running ledger of paid-for labels.

The whole loop is twenty lines. No entropy, no uncertainty, no probabilistic model. The filter is "does this row look like the ones we liked?" and the answer comes from two distx calls.

Now the evaluation question. Does it actually work?

Honest evaluation needs a holdout. Split rows 50/50 into train and test. Run active on train. Build a tree from what it labeled (next chapter). Rank test rows by the tree's predictions, using only their features. Pay for the top check ground-truth labels. Pick the winner. Score that winner against a god-view of the full dataset:

function validate(rs, god, w)
  check = the[:check]
  body  = shuffled(rs[2:end])
  n     = length(body) ÷ 2
  train = vcat([first(rs)], body[1:n])
  test  = body[n+1:end]
  _, lab, labels = active(train)
  y_god = r -> disty(god, r)
  train_best = argmin(y_god, lab.rows)
  tr = tree_grow(lab, lab.rows)
  sorted = sort(test,
    by = r -> mid(tree_leaf(tr, r).ynum))
  top  = sorted[1:min(check, length(sorted))]
  pick = argmin(y_god, top)
  w(train_best), w(pick), labels + check end

Every god-disty call is counted. Total label cost per run is budget + check: 50 + 5 = 55 on defaults. No y-leaks anywhere during training; the tree never sees a god-disty, only the labels active paid for.

The result on auto93.csv: median hold-win around 94 out of 100 at budget=50. For comparison: label 55 rows at random and pick the best gets you around 50. A full-dataset baseline, with access to every label, tops out around 97. Fifty-five labels get us to 97 percent of full-dataset performance.

Try it yourself

Write --active: run validate twenty times with different seeds, print each run's train-win, hold-win, and labels-used. Sort by hold-win and take the median. At budget=50 it should land near 94. At budget=30, with centroid ranking, around 91. With tree ranking, around 85. The six-point gap at thirty labels is the explanation tax, which Chapter 6 is about.

6. Trees and the explanation tax

Chapter 5 ends with "build a tree, rank holdout by leaf-predicted y-value." This chapter builds that tree and counts what it costs.

A decision tree partitions rows into buckets. At each internal node, one column and one cut value. Rows with column-value below the cut go left, above go right. Each leaf is a bucket; its predicted y is the mean of its rows' disty values.

With the stats of Chapter 3 and the distance of Chapter 4 in hand, most of the tree is composition.

@kwdef mutable struct Tree
  d=nothing; ynum=nothing; col=nothing; cut=nothing
  left=nothing; right=nothing; hdr="" end

function new_tree(d, rs, hdr="")
  sub = Data()
  add!(sub, d.cols.names)
  for r in rs; add!(sub, r); end
  ys = Num()
  for r in rs; add!(ys, disty(d, r)); end
  Tree(d=sub, ynum=ys, hdr=hdr) end

A node is a Data (the rows at this node), a Num of their disty values (the node's score), plus branch info and a pre-formatted header for printing.

Choosing cuts on numeric columns is where traditional CART gets expensive and overfits. Standard CART sorts the values and evaluates every candidate threshold, picking whichever minimizes training impurity. With fifty labels that means forty-nine cut candidates per numeric column, and the "best" one is often the one that happened to separate a noise-value from the rest. The optimizer finds spurious structure in sparse data.

We regularize by fiat. For numeric columns, only consider the median:

cand_cuts(::Sym, vs) = unique(vs)
cand_cuts(::Num, vs) = [sort(vs)[length(vs) ÷ 2 + 1]]

One cut per numeric column. A strong prior: we believe the median is good enough regardless of the training labels. Trades theoretical optimality for variance reduction. Works because the trees stay shallow and are only asked to rank rough regions, not make precise predictions.

Greedy split selection over all (column, cut) pairs picks whichever minimizes weighted spread of its children:

function tree_split(d, c, cut, rs)
  l, r = Any[], Any[]; ln, rn = Num(), Num()
  for row in rs
    if go_left(c, cell(c, row), cut)
      push!(l, row); add!(ln, disty(d, row))
    else
      push!(r, row); add!(rn, disty(d, row)) end end
  (ln.n * spread(ln) + rn.n * spread(rn), c, cut, l, r) end

The same spread method handles both regression and classification because Chapter 3's multiple dispatch already resolved the type. Same impurity code for both tasks.

Recurse until each node has fewer than 2*leaf rows. Default leaf=3, so nodes stop splitting at five rows. Print with mid on the left, size in parens, condition on the right:

0.35 ( 27)   Mpg <= 25
0.30 ( 13)   |.. Hp <= 100
0.41 ( 14)   Mpg >  25

Three columns of information: node mean disty, node size, branch condition. Read top to bottom and every decision the tree made is visible. Try that with a transformer.

This is the XAI payoff. Rudin called for it2: high-stakes decisions deserve interpretable models, not post-hoc explanations of opaque ones. Here is an interpretable model, in sixty lines.

But it is not free. We measured the tax. On auto93 at budget=30, ranking the holdout via a tree costs six points of median hold-win versus ranking via a single best centroid. At budget=50 the tax is zero. Above fifty labels the tree actually wins.

The reason is variance. Each leaf's mean disty comes from a handful of rows. With few labels, those means are noisy and the tree routes test rows wrongly. A centroid is a single stable point: less expressive, also less variable. At small N, variance dominates bias. At large N, bias dominates. Trees win when there is enough data to stabilize the leaf estimates.

The takeaway for interpretability advocates: your model has a minimum sample size. Below it, you are paying an explanation tax on noise-amplified decision regions. Above it, the tax vanishes. Build both rankers. Show both numbers. Let the reader pick.

Try it yourself

Write --tree: run active to get a budget of labels, grow a tree from them, print it. Try leaf=3, leaf=5, leaf=7. Smaller leaves give more decision regions and more variance. Larger leaves give fewer regions and more stability. Find your sweet spot for a dataset you care about.

7. Notation

Why Julia? Not because it is shorter than Lisp. Look at three implementations of the same pipeline:

fri.lisp    495 lines   Common Lisp
ezr.jl      375 lines   Julia
ezr.py      543 lines   Python (superset; has kmeans, SA, Bayes)

The fri.lisp-equivalent subset of ezr.py is under four hundred Python lines. Julia is shortest; the three cluster within twenty percent of each other. If you measure bytes of actual code (comments stripped), Julia is about 35 percent fewer bytes than Lisp. The Python subset sits in between.

So why Julia here instead of Lisp or Python? Multiple dispatch as first-class syntax.

function add!(n::Num, v, w=1) ... end
function add!(s::Sym, v, w=1) ... end
function add!(cs::Cols, row, w=1) ... end
function add!(d::Data, row, w=1) ... end

Four methods, one generic function. Julia selects the right one at call time based on the first argument's type. Lisp's defmethod does the same thing. Python's functools.singledispatch is awkward and single-argument, so most Python ports of this code end up with if isinstance(c, Num): ladders inside a single add function. That costs lines; it also costs clarity, because now the function knows about all types rather than each type knowing about itself.

What about Lisp's reader macros ($field for slot access, !key for config lookup)? Those save characters per access, but Julia's n.mu is already shorter than (slot-value n 'mu). The reader-macro advantage shrinks when your base language has fluent field access. Where Lisp does pull ahead is in genuinely custom syntax — reader macros let you invent notation that no language-without-macros can match. For the slice of AI-math work in this book, that capacity doesn't pay.

Python's disadvantage is concrete. Without first-class multiple dispatch, you either write type ladders (verbose, unbranded) or you adopt @singledispatch (awkward decorators, single argument). Either way you write more code per polymorphic operation. Across twenty polymorphic functions, that adds up.

Julia also gives you keyword arguments with defaults (function f(x; y=1)), list comprehensions ([g(x) for x in xs]), @kwdef structs with default values, and JIT compilation to near-C speed. None of these are unique. All of them compound.

You can port ezr.jl to any of the three. The exercise list at the end of each chapter is the port guide. Pick a target. The numbers will be close; the shape will differ.

Try it yourself

Pick a language and port a single function. add! on Num is a good first choice: ten lines here, testable against the Welford formula directly. Count the bytes of your port. That is the cost of the language on this kind of code. Usually within 30 percent either direction of Julia.

8. What we rejected

Along the way we declined two practices most engineers would default to. Both choices matter for the argument.

First, we did not write unit tests. No @test, no test framework, no CI of the usual kind. The file contains nine runnable eg--* demos. --the prints the config. --rows reads and prints a file. --data builds a Data. And so on up to --active, which runs twenty evaluation splits and prints a results table.

Why no unit tests? Because correctness at the unit level is not well-defined for most of this code. What is the correct output of tree_grow on a random five-row subset? The only meaningful test is statistical: across twenty runs at a known seed, does the median hold-win stay near ninety-four? We keep reference implementations — fri.lisp in Common Lisp, ezr.py in Python — that reach the same number. Refactors are validated by running --active on fixed seeds and comparing medians against both. When Julia hits its target, it is correct enough.

This is test-driven development's statistical cousin. The reference implementation is the oracle. The running numbers are the passing bar. It works here because the subject is a learner, not a business rule. If we were validating a billing engine we would have unit tests. Rule of thumb: unit tests for local correctness, statistical evaluation for emergent correctness, pick the right tool for the subject.

Second, we did not document everything. The file has docstrings on a handful of functions, not most of them. The ones with docstrings are the ones a reader would not understand from the name. Everything else has no docstring at all.

Why the silence? Because docstrings on obvious functions decay into noise. Three months after writing "Adds v to the column summary, returning v" on add!, the docstring is useless. New readers learn to skip docstrings entirely, which means the docstrings that actually matter, the ones on non-obvious functions, also get skipped. You lose the leverage of the useful comment by paying for the useless one.

The top-of-file comment block, by contrast, earns its place. It explains the CLI commands, the naming conventions, the CSV format, and the option flags. Fifteen lines of documentation paying for itself every time someone opens the file.

Both choices are instances of the same principle. Do what pays. Skip what does not. Unit tests and document-everything are tools, not rules. Deploy where they earn their place.

Try it yourself

List three practices you consider mandatory in your daily work. For each, name what the practice is paying for, not the name of the principle it claims. Decide whether, for the current project, that payment is worth it. Keep the ones that pay. Drop the ones that do not. Treat the list as a working document, not a creed.

9. Results

On auto93.csv, a 398-row dataset with eight features and three goal columns (Mpg-, Acc+, Weight-), the ezr.jl pipeline:

julia --startup-file=no --compile=min -O0 ezr.jl \
      -s 1 -b 50 --active

runs in about two seconds (mostly JIT startup) and produces twenty evaluation rows. Median train-win: 97. Median hold-win: 94. Label cost per run: 55 (50 from active, 5 from check).

A random baseline (label 55 rows at random, keep the best) gets median hold-win around 50. A full-dataset baseline with access to every label tops out around 97. Fifty-five labels get us to 97 percent of full-dataset performance.

The tree-ranked variant matches the centroid baseline at budget=50 and loses six points at budget=30. The explanation tax is a threshold, not a slope.

The code, across three implementations of the same pipeline:

ezr.jl      375 lines   Julia (primary reference)
fri.lisp    495 lines   Common Lisp
ezr.py      543 lines   Python (superset)

Zero external dependencies beyond the respective runtime. Reproducible under a seed. Julia and Lisp run in milliseconds once warm; Python in tens of milliseconds.

You can read any of them top to bottom in twenty minutes. You can re-implement them in a weekend. The Try it yourself section at the end of every chapter is the porting guide.

Closing

Knuth chose to be on the bottom of things. In an age of LLMs generating code faster than anyone can verify it, someone has to. Read the code. Compile it. Run the numbers. Port it. Own the whole chain. If you cannot see the whole algorithm on one screen, you do not understand it. You trust it. Trust is a bug.

Try it yourself

Clone ezr.jl. Run julia --startup-file=no --compile=min -O0 ezr.jl -s 1 -b 50 --active. Confirm the median. Then start your port. When you finish, come back and compare.
